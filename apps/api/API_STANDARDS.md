# API Standards (Core API Contract)

This document defines the **required** request/response contract for `apps/api` (api-core). It exists to make:

- OpenAPI (`apps/api/openapi.json`) a *true* source of truth
- The generated npm SDK (`packages/core-api-sdk`) strongly typed and stable
- Errors/pagination consistent across all modules

If an endpoint cannot follow these rules (e.g., proxy/gateway passthrough), it must be explicitly classified as an exception in **“Passthrough Endpoints”** below.

---

## 1) Envelope (Required for JSON Business APIs)

All JSON business endpoints MUST return the `ApiResponse<T>` envelope:

### Success

```json
{
  "success": true,
  "data": {},
  "message": "optional"
}
```

Rules:
- `success: true` MUST be present.
- `data` MUST be present (can be `null` when there is no payload).
- `message` is optional and for humans only (clients must not branch on message text).

### Error

```json
{
  "success": false,
  "error": {
    "code": "SOME_ERROR_CODE",
    "message": "Human readable message",
    "details": {}
  }
}
```

Rules:
- `success: false` MUST be present.
- `error.code` MUST be a stable machine-readable string (see section 2).
- `error.message` MUST be a human-readable message.
- `error.details` is optional and should be structured.

### Request ID / Trace

- API MUST return `X-Request-Id` (or equivalent) on every response.
- Body-level requestId is optional; header is the contract.

---

## 2) Error Codes & HTTP Status Mapping

### Standard HTTP statuses

- `400` `BAD_REQUEST`
- `400` `VALIDATION_ERROR` (preferred for schema/body/query validation failures)
- `401` `UNAUTHORIZED`
- `403` `FORBIDDEN`
- `404` `NOT_FOUND`
- `409` `CONFLICT`
- `413` `PAYLOAD_TOO_LARGE` (uploads)
- `429` `RATE_LIMITED`
- `500` `INTERNAL_SERVER_ERROR`
- `502`/`503` reserved for upstream proxy failures (passthrough endpoints)

### Validation error `details` shape (Required when code = VALIDATION_ERROR)

```json
{
  "issues": [
    { "path": "field.or.index.path", "message": "why invalid", "code": "RULE" }
  ]
}
```

Notes:
- `path` is a string path (e.g. `items[0].quantity`).
- `code` is a stable short rule name (e.g. `REQUIRED`, `MIN`, `FORMAT`).

---

## 3) Pagination (Default for All List Endpoints)

All list endpoints MUST use **page/limit** inputs and return `PageResult<T>` inside `data`.

### Request (query)

- `page?: number` (default `1`)
- `limit?: number` (default `10` unless specified)

### Response `data` (PageResult)

```json
{
  "items": [],
  "page": 1,
  "limit": 10,
  "total": 123,
  "totalPages": 13
}
```

Rules:
- Use `items` (not `data`) as the list field inside `PageResult`.
- `totalPages = ceil(total / limit)`
- Endpoints returning collections MUST adopt this format unless explicitly marked as a non-paged exception.

---

## 4) Non-paged Exceptions (Explicitly Allowed Only)

Non-paged list responses are allowed only for small, bounded datasets (e.g., static enums/config options)
and MUST be explicitly documented as an exception in the endpoint description.

### A) `ListResult<T>` (exception when total matters)

```json
{
  "items": [],
  "total": 123
}
```

### B) `T[]` (exception when total does not matter)

```json
[]
```

When using option B, it is still wrapped by `ApiResponse<T[]>`:

```json
{ "success": true, "data": [] }
```

---

## 5) Upload Endpoints (multipart/form-data)

Uploads MUST respond with `ApiResponse<UploadResult>`.

### UploadResult

```json
{
  "filename": "stored-name.webp",
  "originalName": "my-image.webp",
  "size": 12345,
  "mimetype": "image/webp",
  "url": "/uploads/..."
}
```

Rules:
- Request uses `multipart/form-data` with a single file field `file`.
- Enforce size/type limits; on failure use `PAYLOAD_TOO_LARGE` or `BAD_REQUEST` with details.

---

## 6) Passthrough Endpoints (Exceptions)

The following endpoint groups are **passthrough** and are NOT required to return `ApiResponse<T>` on success:

- **Theme App Gateway** (`tags: theme-app-gateway`): `/theme-app/...`
- **Plugin Gateway** (`tags: plugin-gateway`): `/api/extensions/plugin/{slug}/api...`
- **System/Docs** endpoints returning non-business payloads (e.g. `/openapi.json`, health checks)

Rules for passthrough endpoints:
- Success response content-type and shape are defined by the upstream target.
- Errors generated by *our* API SHOULD still use the standard error envelope when possible.
- OpenAPI MUST document the correct content-type for success responses (string/binary/html/json).

---

## 7) OpenAPI Requirements (Critical for npm SDK)

OpenAPI must be sufficiently specific to generate useful types.

### MUST
- Every operation must define:
  - request schema (`params`, `querystring`, and/or `requestBody`) with required fields
  - response schemas for success and expected errors
- For JSON business endpoints, success response schema MUST describe the `data` shape precisely.

### MUST NOT
- Avoid `data: {}` (untyped “any”) except for explicitly passthrough endpoints.

### Recommended patterns
- Define reusable JSON Schemas and register them so OpenAPI can reference them (instead of repeating inline objects).
- Use a reusable `PageResult<T>` schema generator for paging endpoints.

---

## 8) Worked Examples

### Example: Account Profile (read)

`GET /api/account/profile`

Success `200`:

```json
{
  "success": true,
  "data": {
    "id": "usr_123",
    "email": "a@b.com",
    "username": "alice",
    "avatar": "https://...",
    "createdAt": "2026-02-05T00:00:00.000Z",
    "updatedAt": "2026-02-05T00:00:00.000Z",
    "languagePreferences": {
      "preferredLanguage": "en",
      "timezone": "UTC",
      "dateFormat": "YYYY-MM-DD",
      "timeFormat": "HH:mm",
      "numberFormat": "en-US",
      "currencyFormat": "USD"
    }
  }
}
```

### Example: Products List (paged)

`GET /api/products?page=1&limit=10`

Success `200`:

```json
{
  "success": true,
  "data": {
    "items": [
      { "id": "prd_1", "name": "T-Shirt", "price": 19.99 }
    ],
    "page": 1,
    "limit": 10,
    "total": 42,
    "totalPages": 5
  }
}
```

### Example: Validation error

Success `400`:

```json
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid request",
    "details": {
      "issues": [
        { "path": "email", "message": "Invalid email format", "code": "FORMAT" }
      ]
    }
  }
}
```
